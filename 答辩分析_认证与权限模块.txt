================================================================================

【架构】前后端分离，B/S 架构

二、注册模块分析
================================================================================

【涉及文件】
  后端：
    backend/.../controller/AuthController.java          — POST /api/auth/register
    backend/.../service/impl/UserServiceImpl.java        — register() 核心业务逻辑
    backend/.../dto/request/RegisterRequest.java         — 注册请求 DTO
    backend/.../entity/User.java                         — 用户实体映射 sys_user 表
    backend/.../mapper/UserMapper.java                   — MyBatis-Plus 数据访问
  前端：
    frontend/src/views/Register.vue                      — 注册页面组件
    frontend/src/api/index.js                            — authAPI.register() 封装

【注册流程】
  1. 前端 Register.vue 收集表单字段：username / password / name / phone / email
     / age / gender / role（老人=2 或 家属=3）
  2. 前端做基本非空校验后，调用 authAPI.register(form) → POST /api/auth/register
  3. 后端 AuthController.register() 接收 RegisterRequest（@RequestBody，无 @Valid）
  4. UserServiceImpl.register() 执行业务逻辑：
     (a) 用 MyBatis-Plus QueryWrapper 查询用户名是否已存在 → 存在抛 BusinessException("用户名已存在")
     (b) 如果 phone 非空，查询手机号是否已注册 → 存在抛 BusinessException("手机号已注册")
     (c) 如果 email 非空，用正则 ^[A-Za-z0-9+_.-]+@[A-Za-z0-9.-]+$ 校验格式
     (d) 密码用 BCryptPasswordEncoder.encode() 加密后存入
     (e) 角色默认值：request.getRole() != null ? request.getRole() : 2（默认老人）
     (f) 填充 createdAt / updatedAt 为 LocalDateTime.now()
     (g) userMapper.insert(user) 持久化到 sys_user 表
  5. 注册成功后返回 ApiResponse.success("注册成功")，前端跳转到 /login

【关键方法/技术】
  - BCryptPasswordEncoder.encode()：Spring Security 提供的密码哈希算法，自动加盐
  - QueryWrapper：MyBatis-Plus 条件构造器，链式构建 SQL WHERE 条件
  - Pattern.compile()：Java 正则表达式进行邮箱格式校验
  - @TableLogic：MyBatis-Plus 逻辑删除注解，isDeleted=1 表示已删除

【难点】
  - 用户名/手机号唯一性校验需要两次数据库查询，在高并发场景下存在竞态条件
    （当前未加数据库唯一索引 + 应用层同步锁）
  - RegisterRequest 未使用 @Valid/@NotBlank 注解校验，依赖前端校验 → 后端
    校验不足，存在脏数据风险
  - 注册时未校验验证码（与登录不同），存在恶意注册风险
  - 密码强度无校验（无最小长度/复杂度要求）

三、登录模块分析
================================================================================

【涉及文件】
  后端：
    backend/.../controller/AuthController.java          — POST /api/auth/login
    backend/.../controller/CaptchaController.java       — GET /api/auth/captcha
    backend/.../service/impl/UserServiceImpl.java        — login() 核心业务逻辑
    backend/.../service/CaptchaService.java             — 验证码生成与校验
    backend/.../dto/request/LoginRequest.java            — 登录请求 DTO (@Valid)
    backend/.../dto/response/LoginResponse.java          — 登录响应 DTO (Builder模式)
    backend/.../util/JwtUtil.java                        — JWT 令牌工具类
 前端：
    frontend/src/views/Login.vue                         — 登录页面 (含验证码展示与刷新)
    frontend/src/components/auth/AuthForms.vue           — 新版认证表单组件
    frontend/src/components/auth/AuthHeader.vue          — 认证页头部 (Logo)
    frontend/src/components/auth/AuthHero.vue            — 认证页插图
    frontend/src/api/index.js                            — authAPI.login() + captchaAPI.get()

【登录流程（完整链路）】
  一、验证码阶段：
    1. 前端 Login.vue onMounted() 调用 captchaAPI.get() → GET /api/auth/captcha
    2. CaptchaService.generateCaptcha():
       (a) 使用 com.wf.captcha.SpecCaptcha 生成 130×48 像素、4位字符的图形验证码
       (b) 验证码文本转小写后以 "CAPTCHA:{uuid}" 为 Key 存入 Redis，有效期 2 分钟
       (c) 返回 { uuid, captchaImg: Base64图片 } 给前端
    3. 前端将 Base64 渲染为 <img> 标签，uuid 保存到 captchaUuid ref 中
    4. 用户可点击验证码图片触发 fetchCaptcha() 刷新

  二、登录提交阶段：
    1. 前端收集 username + password + captchaCode + uuid，调 authAPI.login()
    2. 后端 AuthController.login() 接收 @Valid LoginRequest（@NotBlank 校验四个字段）
    3. UserServiceImpl.login() 执行：
       (a) captchaService.validate(uuid, captchaCode) — "阅后即焚"校验验证码
       (b) MyBatis-Plus QueryWrapper 按 username 查 sys_user 表
       (c) 用户不存在 → BusinessException("用户不存在")
       (d) passwordEncoder.matches(明文, 密文) 比对密码
       (e) 密码错误 → BusinessException("密码错误")
       (f) jwtUtil.generateToken(userId, username, role) 生成 JWT
       (g) Redis 缓存用户对象：key="user:{userId}", 过期 7200 秒
       (h) 构建 LoginResponse(token, userId, username, name, role, phone) 返回

  三、前端接收阶段：
    1. Login.vue handleLogin() 收到响应：
       (a) 将 token/userId/username/name/role 存入 localStorage
       (b) ElMessage.success("登录成功")
       (c) 按角色跳转：role=1 → /admin/users | role=3 → /family | 其他 → /dashboard
    2. 新版 LoginPage.vue 则通过 Pinia useUserStore.login() 统一管理状态

【验证码"阅后即焚"策略】
  CaptchaService.validate() 的核心机制：
    - 从 Redis 取出验证码文本
    - 若不存在 → "验证码已过期，请重新获取"
    - 比对用户输入（忽略大小写）
    - 无论校验通过与否，均 delete(redisKey) → 一个验证码只能用一次
    - 防止验证码复用攻击

【关键方法/技术】
  - SpecCaptcha(130, 48, 4)：开源验证码库，生成 PNG 图片 + Base64 编码
  - BCryptPasswordEncoder.matches()：BCrypt 密码比对（从密文中提取盐值再哈希比对）
  - Redis StringRedisTemplate：验证码临时存储 + 用户会话缓存
  - @Valid + @NotBlank：JSR-303 Bean Validation，登录参数校验
  - LoginResponse 使用 Lombok @Builder 模式构建

【难点】
  - 验证码维度：图片验证码 + Redis UUID 绑定 + "阅后即焚"是一次性防重放设计
  - Redis 双用途：既存验证码（StringRedisTemplate），又存用户缓存（RedisTemplate），
    需要正确选择序列化方式
  - 密码安全：BCrypt 的盐值嵌入密文中，每次 matches() 需从密文提取盐值重新计算，
    计算耗时约 0.1s，高并发登录场景是性能瓶颈
  - 登录失败时自动刷新验证码并清空输入，提升用户体验

四、角色识别与权限分配
================================================================================

【涉及文件】
  后端：
    backend/.../entity/User.java                         — role 字段定义 (Integer)
    backend/.../config/SecurityConfig.java                — Spring Security 配置
 前端：
    frontend/src/router/index.js                         — Vue Router 路由守卫
    frontend/src/stores/user.js                          — Pinia 用户状态管理
    frontend/src/views/Login.vue                         — 登录后角色跳转
    frontend/src/views/admin/Users.vue                   — 管理员用户管理页

【角色定义体系】
  角色使用 Integer 存储在 sys_user.role 字段：
    角色值    角色名称    前端 Tag 颜色    登录后跳转
    ──────    ────────    ────────────    ──────────
    1         管理员       danger (红)     /admin/users
    2         老人         success (绿)    /dashboard
    3         家属         warning (橙)    /family

  角色映射表（前端两处定义）：
    - Pinia Store user.js: roleLabel computed → {1:'管理员', 2:'老人', 3:'家属'}
    - admin/Users.vue: getRoleText() / getRoleType() → 表格列渲染

【权限分配策略 — 纯前端路由级控制】
  当前项目的权限控制完全在前端 Vue Router 层面实现，后端无接口级鉴权：

  1. 路由守卫 (router.beforeEach):
     - 未登录拦截：除 /login 和 /register 外，所有路由需要 token，否则重定向 /login
     - 角色路由隔离：role===3（家属）访问老人路径时重定向到 /family
       老人专属路径列表：/dashboard, /health, /health-input, /devices,
       /emergency, /services, /orders, /activities, /family-pay

  2. 登录后跳转分发（Login.vue handleLogin）：
     - role === 1 → window.location.href = '/admin/users'
     - role === 3 → window.location.href = '/family'
     - 其他 → window.location.href = '/dashboard'

  3. 管理员用户管理页（admin/Users.vue）：
     - 可查看所有用户列表（含角色标签可视化）
     - 可添加/编辑/删除用户
     - 可为用户充值钱包
     - 角色下拉：管理员(1) / 老人(2) / 家属(3)

【后端安全现状】
  SecurityConfig.java 配置：
    - .antMatchers("/api/**").permitAll() — 所有 API 全部放行
    - 无 JWT 认证过滤器（没有 OncePerRequestFilter 实现）
    - 无 @PreAuthorize / @Secured 方法级权限注解
    - 无 SecurityContext 用户上下文设置
    - 仅依赖前端路由守卫做访问控制

  这意味着：知道 API 路径即可绕过前端直接访问后端接口。

【关键方法/技术】
  - Vue Router beforeEach 导航守卫：全局前置钩子，to/from/next 控制跳转
  - Pinia Composition API (defineStore + setup 函数式)：现代化状态管理
  - localStorage：前端持久化角色信息（token/role/userId 等）
  - Element Plus <el-tag>：角色彩色标签渲染

【难点】
  - 后端无接口级鉴权是最大安全短板：JWT 虽然生成了且包含 role，但后端从未验证
  - 前端路由守卫依赖 localStorage，可被篡改（用户可手动修改 localStorage 中的 role）
  - 角色信息存储在 JWT claims 中但从未在后端被读取和校验
  - 改进方向：需实现 JWT 认证过滤器 + @PreAuthorize 方法级权限控制

五、JWT 令牌机制
================================================================================

【涉及文件】
  后端：
    backend/.../util/JwtUtil.java                         — JWT 核心工具类

【实现细节】
  依赖：com.auth0.jwt (Java JWT 4.4.0)
  配置：application.yml
    jwt.secret = smart-community-care-jwt-secret-key-2024
    jwt.expire = 86400000 (24小时)

  JwtUtil 提供五个方法：

  1. generateToken(Long userId, String username, Integer role)
     - 使用 JWT.create() 构建令牌
     - 载荷 (Payload) 包含三个自定义 Claims：
       · "userId" → userId (Long)
       · "username" → username (String)
       · "role" → role (Integer)
     - 过期时间：System.currentTimeMillis() + expire (当前时间 + 24h)
     - 签名算法：Algorithm.HMAC256(secret.getBytes()) — HMAC-SHA256 对称签名

  2. verifyToken(String token)
     - 使用 JWT.require() 构建验证器，指定相同 HMAC256 算法和密钥
     - .verify(token) 返回 DecodedJWT
     - 若 token 无效/过期/签名不匹配 → 抛出 JWTVerificationException
     - 捕获后包装为 RuntimeException("无效的token")

  3. getUserIdFromToken(String token) — 先 verify 再 getClaim("userId").asLong()
  4. getUsernameFromToken(String token) — 先 verify 再 getClaim("username").asString()
  5. getRoleFromToken(String token) — 先 verify 再 getClaim("role").asInt()

【JWT 在项目中的实际使用情况】
  ★ 生成：登录成功时调用 jwtUtil.generateToken()，将 token 返回前端
  ★ 传输：前端 Axios 拦截器自动在请求头添加 Authorization: Bearer <token>
  ★ 验证：❌ 后端没有任何拦截器/过滤器调用 jwtUtil.verifyToken()
  ★ 角色提取：❌ 后端接口未从 JWT 提取 role 做权限判断

【关键方法/技术】
  - HMAC256 对称签名：签发和验证使用同一密钥，适合单体应用
  - Claims 自定义载荷：在 JWT Payload 中嵌入业务字段
  - DecodedJWT：验证成功后返回的解码令牌对象，可读取 Claims

【难点】
  - JWT 已完整实现生成+验证逻辑，但验证链路未接入 Spring Security 过滤器链
  - 密钥硬编码在 application.yml 中，生产环境应使用环境变量或配置中心
  - 24 小时过期无刷新机制：token 过期后用户必须重新登录
  - JWT 无法主动失效（无状态特性）：即使用户被删除，已签发的 token 仍然有效直到过期
  - 前端 Authorization header 已自动注入，但后端没有对应的解析逻辑 → "穿上了但没锁门"

六、居民档案管理（用户 CRUD）
================================================================================

【涉及文件】
  后端：
    backend/.../controller/UserController.java           — RESTful CRUD 接口
    backend/.../service/UserService.java                  — 业务接口定义
    backend/.../service/impl/UserServiceImpl.java         — 业务实现
    backend/.../entity/User.java                          — 用户实体 (sys_user 表映射)
    backend/.../mapper/UserMapper.java                    — 数据访问层
 前端：
    frontend/src/views/admin/Users.vue                    — 管理端用户列表+CRUD
    frontend/src/views/Profile.vue                        — 个人资料页
    frontend/src/api/index.js                             — userAPI CRUD 封装

【数据模型 — sys_user 表】
  字段              类型            说明
  ─────             ────            ────
  id                Long (自增)     主键
  username          String          用户名（登录凭证）
  password          String          BCrypt 加密密码
  phone             String          手机号
  email             String          邮箱
  name              String          真实姓名
  age               Integer         年龄
  gender            String          性别（男/女）
  address           String          地址
  avatar            String          头像 URL
  role              Integer         角色：1=管理员 2=老人 3=家属
  healthStatus      Integer         健康状态
  emergencyContact  String          紧急联系人姓名
  emergencyPhone    String          紧急联系人电话
  isDeleted         Integer         @TableLogic 逻辑删除标记
  createdAt         LocalDateTime   创建时间
  updatedAt         LocalDateTime   更新时间

【RESTful API 设计】
  GET    /api/users       — 获取所有用户列表
  GET    /api/users/{id}  — 根据 ID 获取单个用户
  PUT    /api/users/{id}  — 更新用户信息
  DELETE /api/users/{id}  — 删除用户（逻辑删除，isDeleted=1）

【业务逻辑】
  - getAllUsers()：无条件查询全部（selectList(null)），无分页
  - getUserById()：selectById 主键查询
  - updateUser()：setUpdatedAt(LocalDateTime.now()) 自动更新时间戳，updateById 更新
  - deleteUser()：deleteById，MyBatis-Plus 自动转为逻辑删除（@TableLogic 配置）
  - 无软删除过滤：getAllUsers() 和 getUserById() 未过滤 isDeleted=1 的记录

【前端展示 — admin/Users.vue】
  - 表格列：ID / 用户名 / 姓名 / 手机号 / 年龄 / 角色(彩色Tag) / 健康状态 / 创建时间 / 操作
  - 操作按钮：编辑 / 充值 / 删除
  - 编辑弹窗：可修改用户名/姓名/手机号/年龄/性别/角色
  - 添加用户：复用注册接口 authAPI.register()
  - 充值弹窗：调用 walletAPI.recharge()，预设快捷金额 10/50/100/200/500

【关键方法/技术】
  - MyBatis-Plus BaseMapper<T>：继承后自动获得 CRUD 方法
  - @TableLogic：逻辑删除注解，deleteById 自动转为 UPDATE SET is_deleted=1
  - QueryWrapper：条件构造器，eq() 精确匹配
  - LocalDateTime：Java 8 时间 API

【难点】
  - 无分页查询：getAllUsers() 全表返回，用户量大时有性能问题
  - 更新接口安全隐患：PUT /api/users/{id} 未校验操作者身份，任意用户可修改他人信息
  - 删除操作为逻辑删除，但查询时未过滤已删除记录
  - 紧急联系人信息直接存在用户表，未做加密处理，存在隐私泄露风险
  - 无数据脱敏：手机号直接明文返回

七、家属关系绑定
================================================================================

【涉及文件】
  后端：
    backend/.../controller/UserRelationController.java   — 关系 CRUD 接口
    backend/.../service/UserRelationService.java          — 关系业务接口
    backend/.../service/impl/UserRelationServiceImpl.java — 关系业务实现
    backend/.../entity/UserRelation.java                  — 关系实体
    backend/.../mapper/UserRelationMapper.java            — 关系数据访问
    backend/.../dto/RelationWithUserDTO.java              — 关系+用户联合 DTO
 前端：
    frontend/src/views/admin/Relations.vue                — 管理端关系管理页
    frontend/src/api/index.js                             — relationAPI 封装
 数据库：
    fix_roles.sql / database_upgrade_v2.sql               — 角色修复 + V2 升级

【数据模型 — user_relation 表】
  字段             类型            说明
  ─────            ────            ────
  id               Long (自增)     主键
  elderId          Long            老人用户 ID（外键 → sys_user.id）
  familyMemberId   Long            家属用户 ID（外键 → sys_user.id）
  relationType     String          关系类型：child(子女)/spouse(配偶)/parent(父母)/sibling(兄弟姐妹)
  status           Integer         状态：1=有效 0=无效
  createdAt        LocalDateTime   创建时间

【RESTful API 设计】
  POST   /api/relations/bind              — 绑定关系
  DELETE /api/relations/{id}              — 解绑关系
  GET    /api/relations/elder/{elderId}   — 查询某老人的所有家属
  GET    /api/relations/family/{familyId} — 查询某家属的所有老人
  GET    /api/relations/exists            — 检查关系是否存在
  DELETE /api/relations/clean-duplicates  — 清理重复关系

【关系绑定核心逻辑 — UserRelationServiceImpl.createRelation()】
  1. 防重复检测：调用 findByElderAndFamily(elderId, familyMemberId) 查询是否已存在
  2. 若已存在：激活关系（status 设为 1），更新记录 —"幂等绑定"
  3. 若不存在：新建 UserRelation 记录，relationType 默认 "child"
  4. 填充 createdAt = LocalDateTime.now()，status = 1

【查询增强 — 关系 + 用户信息联合】
  getRelationsWithUserByElder() / getRelationsWithUserByFamilyMember()：
    - 先查关系表获取 relation 列表
    - 遍历每条 relation，分别查 UserService.getUserById() 获取老人/家属的 name 和 phone
    - 组装为 RelationWithUserDTO 返回
    - ⚠ N+1 查询问题：每条 relation 触发 2 次用户查询

【去重清洗 — cleanDuplicateRelations()】
  使用 JdbcTemplate 直接执行原生 SQL：
    1. SET FOREIGN_KEY_CHECKS=0 — 禁用外键检查
    2. CREATE TEMPORARY TABLE — 创建临时表存每组 (elder_id, family_member_id) 的最小 id
    3. DELETE ... WHERE id NOT IN — 删除不在临时表中的重复记录
    4. DROP TEMPORARY TABLE — 清理临时表
    5. SET FOREIGN_KEY_CHECKS=1 — 恢复外键检查
  标注 @Transactional 保证原子性

【前端关系管理页 — admin/Relations.vue】
  - 统计卡片：老人总数 / 家属总数 / 绑定关系数
  - 老人卡片网格：每个老人一张卡片，展示其下所有绑定家属
  - 家属信息：头像首字母、姓名、关系类型标签、手机号
  - 解绑按钮：每条家属关系可独立解绑（带确认提示）
  - 绑定弹窗：选择老人 + 选择家属 + 选择关系类型

【关系类型映射】
  child   → 子女
  spouse  → 配偶
  parent  → 父母
  sibling → 兄弟姐妹

【关键方法/技术】
  - MyBatis @Select 注解：直接在 Mapper 接口写 SQL
  - JdbcTemplate：绕过 ORM 执行原生 DDL/DML
  - 临时表 (TEMPORARY TABLE)：会话级临时表，连接关闭自动删除
  - @Transactional：Spring 声明式事务
  - 双重查询防重复：先查后插，已存在则激活（幂等设计）

【难点】
  - N+1 查询性能瓶颈：关联查询时逐条查用户信息，老人有 N 个家属时触发 2N 次查询
    优化方向：用 IN 查询批量获取用户，再在内存中 JOIN
  - 去重操作用 JdbcTemplate 绕过 MyBatis-Plus，需手动管理外键约束开关
  - 关系表无唯一约束：虽然代码层面做了防重复，但数据库层缺少 UNIQUE(elder_id, family_member_id)
    约束，极端并发下仍可能产生重复
  - 双向查询：需同时支持"查老人的家属"和"查家属的老人"两个方向
  - 关系类型枚举值（child/spouse/parent/sibling）未在数据库或代码中定义枚举类，依赖
    字符串匹配，存在输入脏数据风险
  - 绑定/解绑无权限校验：任何用户可绑定任意两人

【Pinia Store — stores/user.js】
  采用 Composition API (Setup Store) 风格：

  State (响应式状态，初始化时从 localStorage 恢复)：
    token, role, userId, username, name, phone, avatar

  Getters (计算属性)：
    - isLoggedIn: !!token.value
    - roleLabel: {1:'管理员', 2:'老人', 3:'家属'}[role]
    - roleMap: {1:'/admin', 2:'/elder', 3:'/family'}

  Actions (业务方法)：
    - login({username, password, captchaCode, uuid})：
      调用 authAPI.login() → 更新 state + 写入 localStorage
    - register(form)：调用 authAPI.register() → 成功后前端跳转 /login
    - logout()：清空 state + 清空 localStorage + window.location.href='/login'
    - fetchProfile()：调用 userAPI.getById() 更新本地用户信息

【Axios 拦截器 — api/index.js】
  请求拦截器：
    - 从 localStorage 读取 token
    - 注入 Authorization: Bearer <token> 请求头
    - 实现无感 Token 传递

  响应拦截器（三层错误处理）：
    第一层 — 业务错误 (HTTP 2xx 但 body.code !== 200)：
      → 处理乐观锁冲突、余额不足、权限不足等业务异常
    第二层 — HTTP 错误 (4xx/5xx)：
      · 401 → 清除登录态 → window.location.href='/login'
      · 403 → 返回"权限不足"
      · 其他 → 返回服务端错误信息
    第三层 — 网络错误：
      · ECONNABORTED → "请求超时"
      · 其他 → "网络连接失败"

【Vue Router 路由守卫 — router/index.js】
  路由结构：
    /              → 重定向 /login
    /login         → Login.vue
    /register      → Register.vue
    /dashboard     → 老人首页
    /health        → 健康管理
    /family        → 家属门户 (嵌套路由)
    /admin         → 管理后台 (嵌套路由)

  全局前置守卫 beforeEach：
    1. 检测 token：无 token 且非登录/注册页 → 重定向 /login
    2. 角色隔离：role===3（家属）且访问老人专属路径 → 重定向 /family
    3. 老人专属路径：/dashboard, /health, /health-input, /devices,
       /emergency, /services, /orders, /activities, /family-pay

【难点】
  - Pinia 状态与 localStorage 的双向同步：需要手动维护两套数据一致性
  - 登录后有 Login.vue（直接写 localStorage）和 LoginPage.vue（用 Pinia Store）两套实现
  - 路由守卫从 localStorage 解析 role（parseInt），若被篡改可绕过前端限制
  - 无 token 过期主动检测：只有在 API 返回 401 时才知 token 过期


================================================================================
十、核心难点总结
================================================================================

【难点 1：验证码"阅后即焚"机制】
  问题：如何防止验证码被重复使用？
  方案：Redis 存储 → 校验（对比）→ 无论成功失败都删除 Key → 一次性验证码
  技术点：Redis StringRedisTemplate + UUID 绑定 + 2分钟 TTL 兜底
  价值：有效防止自动化脚本的验证码复用攻击

【难点 2：BCrypt 密码哈希与性能平衡】
  问题：BCrypt 的 bcrypt 哈希 rounds 决定安全性，但 rounds 越高越慢
  方案：使用 Spring Security 默认 BCryptPasswordEncoder（rounds=10）
  影响：单次密码验证约 0.1s，高并发登录场景需要关注
  扩展：可将 rounds 配置化，生产环境调高

【难点 3：JWT 无状态与主动失效的矛盾】
  问题：JWT 签发后无法从服务端主动失效
  现状：即使管理员删除用户，已签发的 JWT 在 24 小时内仍然有效
  改进方向：引入 Redis Token 黑名单机制，删除用户时将 token 加入黑名单

【难点 4：家属关系 N+1 查询问题】
  问题：查询老人→家属列表时，每条关系触发 2 次用户查询
  场景：1 个老人有 5 个家属 = 1(查关系) + 5×2(查用户) = 11 次 SQL
  改进方向：批量查询用户 ID 集合，一次 SQL 获取所有用户，内存中组装

【难点 5：前后端权限分离的架构缺陷】
  问题：前端做路由守卫 + 角色隔离，后端 API 完全无保护
  风险：知晓 API 路径的攻击者可绕过前端直接调用后端接口
  改进方向：
    - 实现 JWT 认证过滤器 (OncePerRequestFilter)
    - 在过滤器中解析 token → 设置 SecurityContext
    - 使用 @PreAuthorize("hasRole('ADMIN')") 保护敏感接口

【难点 6：角色体系的数据一致性问题】
  问题：角色用 Integer 存储，枚举值在前后端多处硬编码
  风险：若需要新增角色或调整角色值，需同步修改：
    - 数据库已有数据（UPDATE SQL）
    - 后端 User 实体注释
    - 前端 Pinia Store roleLabel
    - 前端 Login.vue 跳转逻辑
    - 前端 router 守卫的 elderPaths 列表
    - 前端 admin/Users.vue 角色下拉
  改进方向：使用枚举类 + 后端配置接口 + 前端统一常量

【难点 7：关系去重的技术选型】
  问题：user_relation 表可能存在重复的 (elder_id, family_member_id) 记录
  方案：使用 JdbcTemplate + 临时表 + 原生 SQL 批量清理
  技术：SET FOREIGN_KEY_CHECKS + CREATE TEMPORARY TABLE + DELETE NOT IN + DROP
  权衡：绕过 ORM 直接操作 JDBC，灵活但需要手动管理事务和外键

【难点 8：多角色路由隔离设计】
  问题：三种角色（管理员/老人/家属）看到的界面完全不同
  方案：Vue Router 嵌套路由 + beforeEach 守卫角色判断
  家属路径：/family 作为父路由，子路由包括 dashboard/elder-health/orders/messages
          /activities/services
  管理路径：/admin 作为父路由，子路由包括 users/devices/emergencies/services
          /orders/relations
  老人路径：顶层平铺 /dashboard /health /services /messages 等

【难点 9：前端跨组件认证状态同步】
  问题：多个组件需要知道当前用户信息（导航栏、个人中心、权限判断等）
  方案：Pinia Store 集中管理 + localStorage 持久化 + API 响应拦截器 401 统一处理
  优势：组件无需各自管理认证状态，一个 logout() 清理所有
